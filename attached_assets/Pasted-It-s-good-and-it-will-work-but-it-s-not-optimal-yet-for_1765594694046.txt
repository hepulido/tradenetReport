What I would change (so it’s “optimal”)
1) Don’t start with 2 separate “labor_entries + transactions” unless you must

In construction, labor can come from:

payroll summaries

time tracking apps

manual timesheets

QB payroll (sometimes not even connected)

For MVP, keep it simple:

✅ Keep transactions for costs/revenue
✅ Keep labor_entries only if you’re sure you’ll track labor hours now
➡️ Otherwise, handle labor as a transaction category (“labor”) until later.

Why: clients won’t give clean labor inputs at first. You need flexibility.

Recommendation: Keep labor_entries, but treat it as optional and don’t build logic that depends on it.

2) You need an “import staging” table (this is huge)

Right now you log imports, but you don’t have a place to store raw rows before mapping.

Add:

import_files (file metadata)

import_rows (raw CSV rows JSON)

mappings (rules to map memo/vendor/category → project/category)

This is what makes it feel “automatic” even when it’s semi-manual.

Without this, you’ll constantly keep doing manual cleanup outside the app.

3) You need “clients trust” features: Auth + RLS

If you want companies to trust you with QuickBooks later, you need to say:

“We use Supabase Auth”

“Each business can only access their data via Row Level Security”

“We never store QB passwords; only OAuth tokens; encrypted”

Even if you’re not implementing QB today, the architecture should support that.

4) Reports: store both “snapshot summary” + “line items used”

Your current weekly_reports(summary jsonb) is good, but I’d also store:

which transactions/labor entries were included (ids)

so the report is auditable (“why is this number here?”)

That’s a very “real product” feature.
